# MVP 继续拆解（2026-03-30）

## 1. 当前实现快照

基于现有代码与测试结果，项目已经形成“后端 API + 局部前端页面”的可运行骨架，当前测试通过（6/6）。

已具备能力（概览）：

- 站点管理：增删改查、规则配置、测试抓取（样例结果）。
- 抓取任务：按关键词创建任务、按站点执行、去重、入库、日志记录、重试。
- 素材库：列表筛选、手动上传、标签统计、素材状态操作。
- 文章生成：预览、保存草稿、编辑、复制、删除、发布结构化数据生成。
- 草稿发布：已具备“模拟微信发布链路”（sync/publish）。
- 历史统计：抓取/素材/发布/日志聚合接口。

## 3. “继续拆解”优先级任务清单

## P0（必须先做，直接影响 MVP 可交付）

### P0-1：补齐四个前端模块页面模板

目标：打通 `materials / articles / drafts / history` 页面入口和交互承载。  
交付：
- 新增 4 个模板文件并接入导航与脚本。
- 页面元素 ID 与现有 `app.js` 对齐，确保无需大改 JS 即可运行。
- 完成模块级手工验证（列表、按钮、关键操作提示）。

### P0-2：抓取服务从“模拟结果”升级为“可插拔真实抓取”

目标：将站点抓取从内置模拟数据改为适配器驱动。  
交付：
- 定义抓取适配器接口（按站点 code 分发）。
- 落地至少 1 个真实可用站点适配器（可先用公开 API 站点）。
- 保留 mock 兜底模式（开发环境可切换）。
- 明确超时、重试、错误日志与去重行为。

### P0-3：微信发布链路接入真实 API（保留 mock）

目标：完成“可真实发布”的工程闭环。  
交付：
- 抽象微信客户端层（token 获取/刷新、素材上传、草稿创建、发布调用）。
- 引入配置项控制 `mock/real` 模式切换。
- 发布记录写入真实返回值（含失败原因）。
- 对敏感失败场景做可重试处理。

## P1（MVP 增强，建议紧随 P0）

### P1-1：素材库“爬取记录视图”补全

目标：把“按名称/素材编号查记录”从“能查到”升级为“可视化可追踪”。  
交付：
- 在素材详情区显示 crawl_task_id、来源站点、关键词、任务日志摘要。
- 提供跳转到任务详情的入口。

### P1-2：任务执行异步化 + 进度查询

目标：避免长任务阻塞请求，提升可用性。  
交付：
- 创建任务即返回 task_id。
- 后台执行并提供进度轮询接口（后续可扩展 WebSocket）。
- 前端支持“执行中状态刷新”。

## P2（稳定性与可维护性）

### P2-1：分层日志与配置治理

目标：提高排障效率与生产可运维性。  
交付：
- 抓取日志、业务日志、系统错误日志分层输出。
- 敏感配置迁移至环境变量并规范化。

### P2-2：测试覆盖补齐（重点是端到端路径）

目标：防止回归。  
交付：
- 新增页面可用性与关键流程 API 测试。
- 覆盖“真实抓取失败回退”“微信发布失败重试”等关键路径。

## 4. 建议执行顺序（两周节奏）

### 第 1 周

1. P0-1 页面模板补齐（含联调）
2. P0-2 适配器接口 + 首个真实站点
3. 完成回归测试并冻结接口

### 第 2 周

1. P0-3 微信真实发布接入（保留 mock）
2. P1-1 素材爬取记录视图
3. P1-2 任务异步化（最小可用版）

---

## 5. 里程碑验收标准

- 页面层：六大模块页面全部可访问且核心交互可用。
- 抓取层：至少一个真实站点可稳定抓取入库，失败可重试可定位。
- 发布层：支持 mock/real 双模式；real 模式可完成一次真实发布闭环。
- 质量层：核心测试用例通过，关键失败路径可复现可观测。

## 6. 本次拆解结论

当前项目已完成 MVP 的**主体骨架和大部分 API 能力**，下一阶段应从“功能存在”推进到“业务可交付”，优先完成：

1. 前端四个缺失模块页面补齐。
2. 抓取与微信发布从 mock 过渡到真实可切换实现。
3. 素材记录可追踪与任务异步化，提升运营可用性。
